# Progress

## Plan Notes

### Research findings

- **Cloudflare `x-session-affinity`** (Workers AI Prompt Caching): per-request header carrying a unique session/agent ID. Routes the request to the same model instance that holds the cached prefix tensors, raising prefix-cache hit ratio (Cloudflare reports 60% → 80% internally). Available on REST and the Workers AI binding. No documented lifetime, fallback, or privacy constraints. Cloudflare's OpenAI-compatible endpoint accepts the same header transparently.
- **AI SDK abstraction**: no first-class "session affinity" primitive. AI SDK exposes per-call `headers` and `providerOptions`; AI Gateway's `caching: 'auto'` is a cache-control concept (cache_control breakpoints), orthogonal to routing affinity. So the cross-provider general form is simply "per-request HTTP header at the AI-provider boundary".
- **Shrimp current shape**: Shrimp Agent + SummarizePort both speak OpenAI-compatible (`OPENAI_BASE_URL`/`OPENAI_API_KEY`). ChannelJob already has a stable Session ID; HeartbeatJob has no conversation identifier.

### Design intent for the SPEC change (revised after user review)

- **No new vocabulary.** Reuse the existing **Session ID** term — Cloudflare's "session affinity" is just a transport detail. Calling it `Affinity Key` would invent jargon and pull a Cloudflare-flavored name into the use-case layer.
- **Layering.** Use case already has Session ID in scope (ChannelJob path). The SPEC change does NOT need to re-plumb Session ID through ports — it only needs to add a Prompt Caching description stating: at the AI-provider boundary, Shrimp forwards the active Session ID as an HTTP header to enable prefix-cache routing. Cloudflare `x-session-affinity` is the motivating example; other providers ignore the header harmlessly.
- **HeartbeatJob also sends the header (corrected after user review).** Earlier I claimed HeartbeatJob should not send the header because it has no Session — that was wrong. The Shrimp Agent uses `ToolLoopAgent`, so a single Job invocation makes N round-trips against the AI provider, all sharing the same system prompt + tool definitions + initial user prompt. That within-invocation prefix is the highest-frequency, most reliable prefix-cache target. Cloudflare's docs say "unique identifier for your session **or agent**" — affinity is per "request group sharing a prefix", not per conversation. **Identifier source per Job kind:** ChannelJob → **Session ID** (also extends affinity across conversation turns, an additional benefit); HeartbeatJob → **a stable per-invocation Job identifier** (covers the within-invocation round-trips). Naming-wise, both are "the identifier for the current Agent invocation" — no new top-level term required.
- **No environment variable (corrected).** The header name is an infra-internal constant tied to the provider protocol (`x-session-affinity` for Cloudflare). It's not an operator-tunable knob — there's no realistic reason to disable it or change the name at runtime. Adding an env var would only inflate the configuration surface (env-config.ts + `.env.example` + docker-compose.coolify.yml + Coolify UI) for a value nobody needs to touch. Feature is always on; header name is hardcoded in infra.
- **Scope discipline.** Spec-only change, additive: a Prompt Caching subsection. NO new env var, NO new port, NO new failure-handling rules, NO observability changes. Do NOT touch `infrastructure/ai/` or `env-config.ts` in this loop.
