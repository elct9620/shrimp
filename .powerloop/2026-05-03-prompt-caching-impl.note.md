### Codebase findings

- **Wiring point**: `src/infrastructure/ai/ai-sdk-shrimp-agent.ts` — `AiSdkShrimpAgent.buildToolLoopAgentOptions(input)` returns a `ToolLoopAgentSettings`. The AI SDK's `ToolLoopAgentSettings` accepts a `headers?: Record<string,string>` field (verified via `node_modules/ai/dist/index.d.ts` L3336). These headers are forwarded as HTTP request headers on every round-trip the `ToolLoopAgent` makes, including through `@ai-sdk/openai-compatible` (verified: `LanguageModelV3.doGenerate` accepts `headers` per call). This is the natural injection point — single line addition in `buildToolLoopAgentOptions`.
- **Identifier already in `JobInput`**: `src/use-cases/ports/shrimp-agent.ts` already has `jobId: string` (always set) and `sessionId?: string` (set only by ChannelJob). No port change needed; the header value is `input.sessionId ?? input.jobId`.
- **HeartbeatJob source** (`src/use-cases/heartbeat-job.ts`): generates `jobId = randomUUID()` once at top of `run()` and passes via `JobInput.jobId`. Stable per-invocation. Already wired.
- **ChannelJob source** (`src/use-cases/channel-job.ts`): passes `sessionId: session.id` in addition to `jobId`. Already wired.
- **SummarizePort** (`src/infrastructure/ai/ai-sdk-summarize-port.ts`): uses bare `generateText` with no `headers` field. Per SPEC §Prompt Caching the header MUST NOT be sent here — current code already does not send it. Just needs a negative test to lock that in.
- **Provider construction** (`src/container.ts` L178-182): `createOpenAICompatible({ name, baseURL, apiKey })`. The OpenAI-compatible provider supports a top-level `headers` option (constant headers on every request) but we want a **per-call** header derived from per-invocation identifier, so we plumb at the `ToolLoopAgentSettings` layer (per-call), not at provider construction.
- **Header constant**: should live next to the agent file. Naming candidate: `SESSION_AFFINITY_HEADER = "x-session-affinity"` exported from `ai-sdk-shrimp-agent.ts` (or a sibling module) for test reuse.
- **Test conventions**: `tests/infrastructure/ai/ai-sdk-shrimp-agent.test.ts` uses `MockLanguageModelV3` from `ai/test`. The mock receives the call options including `headers` — assertion can be done by capturing the `headers` argument passed into `doGenerate`. For SummarizePort negative test, mirror via `tests/infrastructure/ai/ai-sdk-summarize-port.test.ts` and assert no `x-session-affinity` key.

### Decomposition rationale

- 5 items keeps each cycle small and aligned with the loop's 1-execute-per-cycle model.
- Item 1 (the wiring) is the load-bearing change; 2 and 3 are proof-of-correctness via tests at the unit boundary; 4 is the negative test for SummarizePort independence; 5 is the integration confirmation in container/end-to-end shape.
- HeartbeatJob and ChannelJob ID-sourcing are NOT separate items because both code paths already set `jobId` / `sessionId` on `JobInput`; the agent reads the right one via `sessionId ?? jobId`. The two Job-kind behaviors are differentiated by tests in items 2 and 3, not by code changes in two places.
