# Remove undici Dependency

## Context Snapshot (Plan Phase)

**Audit results:**

- Direct `undici` imports in our code: **none**. All matches in `src/` are documentation comments only.
- `node-fetch@3.3.2` already a direct dependency.
- `@doist/todoist-sdk@9.0.0` lazily imports `undici`'s `EnvHttpProxyAgent` for *every* Node call via `getDefaultDispatcher()` (not just when proxy env vars are set). This is the actual risk path.
- `TodoistApi` constructor accepts `{ customFetch }` option — passing `node-fetch` bypasses the undici dispatcher entirely.
- `telegram-agent.ts` already uses `node-fetch + https.Agent` (commit c70dae9).

**Scope decision (small):** inject `node-fetch` as `customFetch` into `TodoistApi` at the DI wiring point. Verify with a unit test that the SDK never falls back to global fetch / undici dispatcher.

**Out of scope:** forking todoist-sdk, adding pnpm overrides, replacing the SDK with raw HTTP calls. The transitive `undici` entry in `pnpm-lock.yaml` stays (todoist-sdk lists it as a dependency); what matters is that runtime never loads it.
